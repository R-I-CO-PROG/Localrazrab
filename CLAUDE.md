# Инструкции для AI-агентов, работающих с этим проектом на боевом сервере

Прочти это ПЕРЕД тем, как что-либо менять на сервере (`mercai.ru`, `/var/www/Mercai-v2`).
Это боевой сервер с живыми пользователями. Правила ниже появились после реальных инцидентов.

## Главное правило: git — сначала, прод — потом

**Никогда не редактируй файлы напрямую на сервере через SSH, не закоммитив их.**

Правильный порядок для любого изменения кода:
1. Внеси изменение (неважно, локально или прямо на сервере через SSH — технически возможно и то, и то).
2. Сразу закоммить: `git add -- <файлы> && git commit -m "..."`.
3. Запушь: `git push origin main`.
4. Только после этого — пересобери и передеплой (`pnpm run build`, `pm2 restart ...`).

Если правишь несколько несвязанных вещей подряд — коммить их **отдельными** коммитами по смыслу,
не одним огромным. Если находишь на сервере уже существующие незакоммиченные изменения —
**не удаляй и не перезаписывай их не глядя**. Сначала пойми, что это (`git diff`, `git status`),
и если это не твои изменения — предположи, что это чья-то незавершённая работа, и обращайся с ней
бережно (закоммить отдельно как WIP, а не смешивать со своим).

**Почему это важно**: к началу июля 2026 на сервере накопилось ~4930 незакоммиченных строк
изменений (часть — забытая чужая работа над embeddings/neural-selector и админ-панелью,
часть — большая директория с товарными фото, которую давно удалили с диска, но не из git).
Из-за этого `git reset --hard` в деплой-скрипте был реальной угрозой — мог в любой момент
стереть чьи-то правки без предупреждения. Деплой-скрипт (`scripts/deploy-mercai-v2-vps.sh`)
теперь **отказывается запускаться**, если в рабочей директории есть незакоммиченные изменения,
и использует `git merge --ff-only` вместо `reset --hard` — но это подстраховка на крайний случай,
а не повод полагаться на неё вместо нормального git-флоу.

## Особенность этого репозитория: ДВЕ Prisma-схемы, ОДИН сгенерированный клиент

`apps/api/prisma/schema.prisma` и `apps/web/prisma/schema.prisma` — это два РАЗНЫХ файла схемы,
но `prisma generate` из любого из них пишет в **один и тот же** общий `node_modules/.pnpm/@prisma+client.../`
(так устроен pnpm-монорепо). Если схемы разойдутся (в одной есть модель/поле, в другой нет),
то какая из них сгенерировалась последней — та и победит, и приложение, зависящее от
отсутствующей модели, начнёт падать с `TypeError: Cannot read properties of undefined`.

Хуже того: у `apps/api` в `package.json` есть `"prestart": "prisma generate"` — то есть
**просто `pm2 restart mercai-api`** тихо перегенерирует общий клиент из схемы API, даже если
ты только что аккуратно сгенерировал его из схемы Web.

Из-за этого 6 июля 2026 был реальный инцидент: `prisma db push --accept-data-loss` из схемы Web
(в которой не было полей `Product.widthCm/heightCm/depthCm/weightG`, добавленных ранее только
в схему API) удалил эти колонки вместе с данными примерно у 50 тысяч товаров. Данные восстановили
через `apps/api/prisma/backfill-dimensions.ts`, но могло не повезти.

**Правило**: обе схемы должны содержать одинаковый набор моделей (сейчас это так — сверено
7 июля 2026). Если добавляешь модель/поле в БД — добавь его в ОБЕ схемы, даже если конкретное
приложение им не пользуется. Перед любым `prisma db push` или `--accept-data-loss` — сначала
`git diff apps/api/prisma/schema.prisma apps/web/prisma/schema.prisma`, чтобы убедиться, что
разница — только в форматировании, а не в составе моделей.

После правки схемы — перегенерируй клиент **и** синхронизируй standalone-копию Next.js
(`apps/web/.next/standalone/node_modules/.pnpm/@prisma+client.../`), либо просто пересобери
`apps/web` полностью (`pnpm run build` уже прогоняет `prisma generate` сам).

## Деплой и бэкапы

- Полный деплой: `scripts/deploy-mercai-v2-vps.sh` (идемпотентный, безопасен для повторного запуска).
- Бэкап (БД + .env + uploads + состояние PM2): `scripts/backup-mercai-v2-vps.sh`, запускается
  автоматически по cron каждый день в 03:00 (см. `crontab -l`), хранит последние 7 архивов
  в `/var/backups/mercai/`.
- `uploads/` — не в git (см. `.gitignore`), но и не должен быть: он не трогается ни деплоем,
  ни `git merge --ff-only`. Бэкапится отдельно, см. выше.

## Прочее

- `.env` файлы никогда не должны попадать в git — уже в `.gitignore`, так и оставь.
- `API_SECRET_KEY` защищает внутренний API (web BFF → NestJS). Guard в
  `apps/api/src/security/api-key.guard.ts` сейчас пропускает запросы без проверки, если
  переменная вообще не задана (`if (!secret) return true`) — это fail-open, потенциально
  стоит поправить на fail-closed, если будешь трогать этот файл.
